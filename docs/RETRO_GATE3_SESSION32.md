# ideamemo GATE3 振り返り（Session 32）

## 問題と自動化障壁

### 1. 仕様書の手戻りが多すぎる
- グラフ画面だけで v4 → v5（Level 1→2→3）→ v5.1（フラットマップ）の3回書き直し
- v5のClaude Code実装（§1-§5）が丸ごと無駄に
- 原因: 「設計を固めてから実装」か「触ってから変える」かが曖昧
- **自動化不可（人間の体験判断）。** ただし「差分だけ実装する」仕組みで再実装コストは下げられる

### 2. 仕様書とコードの乖離が追跡できていない
- v4/v5/v5.1/FEEDBACK/SPEC_GRAPH_HOME等が乱立。どれが「現在の正」か不明
- Claude Codeが古いspecを参照するリスク
- **対策:** CLAUDE.md内に「現在有効なspecとセクション」一覧を管理 + 古いspecに SUPERSEDED マーク

### 3. UI検証が目視のみ
- tsc/buildは通ってもUIの意図通りかはShotaの実機目視のみ
- 構造検証（要素の存在、接続線の本数等）はE2Eで自動化可能だがゼロ
- **対策:** Playwright E2E導入（構造検証のみ。「気持ちよさ」は目視のまま）

### 4. 設計チャットとClaude Codeの役割分担が曖昧
- このチャット = 設計議論 + spec執筆 + Claude Code指示書作成が混在
- 設計議論の結論 → spec差分 → 指示書の変換が毎回手動
- **対策:** n8nパイプライン（Slackでの決定事項 → spec差分変換 → Claude Code起動）

### 5. PATトークン運用
- 毎セッション同じトークンを貼り続けている。revoke忘れリスク常在
- **対策:** GitHub Appインストール型認証に切替。このチャットからのgit pushを不要にする

## 優先度マトリクス

| 問題 | 影響度 | 自動化難易度 | 優先度 |
|------|--------|-------------|--------|
| spec手戻り | 大 | 不可（人間判断） | 運用で対処 |
| spec乖離の追跡 | 中 | 低（メタデータ追加） | ★高 |
| UI検証が目視のみ | 大 | 中（E2E導入） | ★高 |
| 設計→指示書の変換 | 中 | 中（パイプライン構築） | 中 |
| PAT運用 | 小 | 低（GitHub App化） | 低 |
